## Context

### 背景与现状

- 当前项目已有一套基于 Git 仓库的「城市」可视化能力，主要面向 GitHub / 单人或单仓库视角，UI 语言以英文为主。
- 团队实际工作场景为：10 多名工程师，分布在约 5 个 GitLab Group、20 个项目中，所有代码托管在公司内网 GitLab，仅在内网环境访问与部署。
- 现状痛点：
  - 无法在一个统一视图中看到团队整体的编码活跃度、协作网络以及项目热度分布。
  - 城市可视化与团队真实组织结构（Group/项目/小组）结合不够紧密。
  - UI 语言为英文，不够贴近日常中文办公语境，降低了日常使用频率。

### 目标环境与约束

- 部署环境：
  - 仅部署在公司内网机器，访问范围为内网（不暴露到公网）。
  - 运行模式沿用现有应用（例如：Next.js / Node 服务 + 前端静态资源），通过内网域名访问。
- GitLab 接入约束：
  - 只能访问公司内网 GitLab 实例，通过 Access Token / OAuth / SSO 获取权限。
  - 需要尊重 GitLab 的权限体系，不展示用户无权访问的项目与数据。
- 语言与交互：
  - 整体 UI 文案、菜单、图例、错误提示等需要改造为中文显示。
  - 需要在不破坏现有英文结构的前提下，引入完整的 i18n 方案，保留未来切回英文或多语言的可能性。

## Goals / Non-Goals

**Goals:**

- 提供一个面向团队视角的「GitLab 城市」：
  - 聚合约 5 个 Group、20 个项目的数据，以城市形态呈现团队整体的活跃度与贡献分布。
  - 支持按成员、Group、项目等维度进行筛选与切换视图。
- 为每个成员提供「个人 GitLab 城市」视图：
  - 输入 GitLab 用户名后，展示该成员在所有纳入统计项目上的综合城市（含编码、Review、Issue 等行为）。
- 建立统一的贡献度/活跃度度量模型：
  - 将 commit、MR、Review、Issue 等行为统一折算为可视化指标（楼高、亮度、体量等）。
- 完成 UI 的中文本地化与 i18n 方案：
  - 核心交互（菜单、提示、过滤条件、图例说明）全部支持中文显示。
  - 确立 i18n 技术方案，便于后续扩展多语言。
- 保持部署与运维简单：
  - 在同一代码仓下扩展功能，仍然可以通过单一服务在内网独立部署。

**Non-Goals:**

- 不在本次设计中引入复杂的「绩效/KPI 打分系统」，贡献度仅用于可视化参考与趋势观察。
- 不覆盖所有 GitLab 实例，仅针对公司内网指定的一个 GitLab 部署。
- 不在本阶段实现跨团队对比/排名等功能，避免引发过度竞争或误读数据。
- 不强制重写现有城市渲染引擎，只在其基础上进行扩展与适配。

## Decisions

### 1. 总体架构与数据流

- **架构风格：**
  - 保持现有前后端一体化架构（假设为 Next.js / Node），在同一服务内新增 GitLab 数据采集与聚合模块。
  - 前端负责城市渲染与交互（包括中文 UI 与过滤器），后端提供统一的城市数据 API。
- **数据流基本链路：**
  1. 后端定期/按需调用 GitLab API，拉取指定 Group/Project 的 commit、MR、Review、Issue 数据。
  2. 将原始事件转换为统一的「贡献事件」模型（如：`CommitEvent`、`MergeRequestEvent`、`ReviewEvent`、`IssueEvent`）。
  3. 在服务内存或持久层中聚合这些事件，计算成员与项目维度的指标（贡献度、活跃度、协作关系）。
  4. 前端通过 REST/GraphQL API 获取聚合后的城市数据，按约定的 schema 渲染成城市几何元素。

**替代方案对比：**

- 方案 A：前端直接调用 GitLab API 按需拉取  
  - 优点：实现简单、无额外后端存储。  
  - 缺点：耦合 GitLab API，受限于浏览器 CORS 与权限，性能与速率不稳定。  
- 方案 B（采纳）：后端统一代理与聚合  
  - 优点：可控的调用频率、统一缓存、可做复杂指标计算，前端只关注展示。  
  - 缺点：增加一定后端实现复杂度。  

决定采用 **方案 B**：后端统一数据采集与聚合。

### 2. GitLab 接入与配置

- **接入方式：**
  - 使用 Personal Access Token 或项目级 Token，通过环境变量注入（如 `GITLAB_BASE_URL`、`GITLAB_TOKEN`）。
  - 在配置文件中维护需要统计的 Group/Project 列表（支持按 Group ID、Project ID/Path 配置）。
- **安全性与权限：**
  - 服务仅部署在内网，对外不暴露 Token。
  - Token 权限尽量最小化，只授予读取项目和仓库活动所需的权限。
- **配置管理：**
  - 使用 `.env.local` 或类似机制管理敏感配置。
  - 为未来在管理后台可视化配置 Group/Project 预留接口层（当前阶段配置走文件/环境变量）。

### 3. 贡献度与活跃度指标模型

- **统一事件模型：**
  - 将来自 GitLab 的不同事件统一建模为内部事件类型：
    - `CommitEvent`：包含 author、project、添加/删除行数、时间戳等。
    - `MergeRequestEvent`：包含创建者、参与者、变更规模、状态（merged/closed）、时间等。
    - `ReviewEvent`：以 MR 评论、审批为主，表示 review 行为。
    - `IssueEvent`：包含创建、关闭等操作，代表需求/缺陷处理行为。
- **加权策略（示意，可在 specs 细化）：**
  - Commit：基础权重按提交次数与变更规模折中，避免单纯以行数排序。
  - MR：合并成功的 MR 给与显著权重，体现“功能交付”。
  - Review：通过评论次数、审批记录给一定加权，鼓励代码评审行为。
  - Issue：为创建/关闭 Issue 赋予一定权重，体现需求与质量工作。
- **输出指标：**
  - 成员维度：
    - 总贡献度（用于楼高）。
    - 项目覆盖广度（参与的项目数，用于楼体占地或簇数量）。
    - 近期活跃度（近 7/30 天贡献度，用于楼体亮度/动画）。
  - 团队维度：
    - Group/项目活跃度热力（用于城市子区域的整体高度或色块强度）。
    - 协作网络强度（用于成员之间连线的粗细/亮度）。

### 4. 城市可视化与视图设计

- **城市布局：**
  - 顶层：按 Group 或业务域划分为多个「街区」，每个街区内按项目布置楼宇群。
  - 成员视角：在「个人城市」中，以成员为中心聚合多个项目楼宇，可按项目或行为类型拆分子楼。
- **交互视图：**
  - 团队主视图：
    - 默认展示最近 30 天的团队城市。
    - 左侧或顶部提供过滤器：时间范围、Group、项目、技术栈标签等。
    - 悬停/点击楼宇显示成员/项目详情（中文信息）。
  - 个人视图：
    - 顶部输入框输入 GitLab 用户名，跳转到个人城市视图。
    - 提供「返回团队视图」入口。
- **协作网络呈现：**
  - 在团队视图中，用线条或光束连接存在 Review/MR 往来的成员楼宇。
  - 支持根据连接强度调整透明度或粗细，避免信息过载。

### 5. 本地化与多语言支持

- **技术方案：**
  - 如果当前为 Next.js/React 栈，采用例如 `next-i18next` / `react-intl` / 自建 i18n hook 的方案。
  - 所有文案（菜单、按钮、tooltip、图例、错误提示等）统一抽离到多语言资源文件（如 `locales/zh-CN.json`）。
- **默认语言：**
  - 内网部署默认语言为中文（简体），URL 上不强制携带语言前缀。
- **兼容性：**
  - 保留英文语言包占位（如 `en.json`），便于未来需要时快速打开英文版本。
  - 在组件中禁止写死字符串，全部通过 i18n key 渲染。

### 6. 部署与运行模式

- 与现有应用共用同一部署流程（例如 CI/CD pipeline 不变）。
- 通过环境变量区分不同环境（如测试环境与正式环境的 GitLab 地址与 Token）。
- 内网访问通过 Nginx/Ingress 等反向代理到 Node/Next 服务。

## Risks / Trade-offs

- **GitLab API 速率与稳定性风险**
  - 风险：多项目、多 Group 聚合时，可能触达 GitLab API 速率限制或在高峰期响应变慢。
  - 缓解：引入请求节流与本地缓存，尽量通过定时任务预拉取数据，在前端请求时命中缓存。

- **贡献度模型被误读为绩效考核工具的风险**
  - 风险：成员可能担心城市高度等同于绩效，影响正常协作氛围。
  - 缓解：在产品文案与团队宣导中强调「城市用于可视化协作状态与节奏，不作为绩效考核依据」；避免展示过于“排行榜化”的 UI。

- **可视化信息过载**
  - 风险：在一个视图上承载过多维度（Group、项目、成员、协作线、风险提示），用户难以理解。
  - 缓解：通过视图分层与开关控制（例如可以隐藏协作线/风险标记），默认为简洁视图，逐步揭示更多信息。

- **本地化维护成本**
  - 风险：后续新增功能若没有遵守 i18n 约束，容易出现中英混杂。
  - 缓解：在代码评审规范中加入「不得直接写死 UI 文案」的校验；必要时引入简单的 lint 规则或 PR checklist。
