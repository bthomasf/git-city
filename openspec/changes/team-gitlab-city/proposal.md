## Why

当前团队有 10 多名成员，分布在约 5 个 GitLab Group、共 20 个左右的项目中，代码分布分散、协作关系复杂，缺少一个「一眼看懂团队整体编码健康度与活跃度」的统一视图。现有 GitHub 城市方案以单人/单仓库为主视角，无法很好地承载多 Group、多项目、多角色的团队视角，也无法在一个「城市」中直观体现团队协作网络、项目热度与个人贡献画像。

因此，我们希望基于现有 Git city 能力，升级为适配 GitLab 的「团队 GitLab 城市」，既能实时/准实时展示团队整体的代码城市风貌，又能按成员、Group、项目灵活切换视角，帮助管理者与一线工程师更好地理解团队的投入结构、协作质量与长期趋势。

## What Changes

- **从 GitHub 个人视角升级到 GitLab 团队视角**
  - 引入「团队城市」主视图：一屏展示团队所有成员的大楼和 Activity 概况。
  - 支持基于 GitLab Group / Project / 自定义团队标签的分组视图切换。
  - 支持在输入框中输入 GitLab 用户名，快速切换到该成员的「个人城市」视图。

- **支持多 Group、多项目的数据聚合与归一化**
  - 对接 GitLab API，从约 5 个 Group、20 个项目中统一采集 commit、MR、Review、Issue 等数据。
  - 通过统一的「贡献度指标模型」对不同项目、不同语言的贡献进行归一化，避免单纯以行数/提交数评估。
  - 引入可配置的「项目权重」与「非代码工作加权」（如 Review、Issue 管理），更贴近互联网团队协作习惯。

- **城市可视化升级：从单仓库楼宇到团队城市地图**
  - 每个成员对应一栋或一簇「主楼」，高度代表整体贡献度，体积代表参与项目广度，亮度/纹理代表近期活跃度。
  - 不同 Group/项目以「街区」或「区域」形式展现，可通过颜色或地块边界区分。
  - 支持基于时间维度（近 7 天 / 30 天 / 90 天 / 自定义区间）对城市进行动态重绘，看到团队节奏变化。

- **个人视角：可输入用户名查看「个人 GitLab 城市」**
  - 输入 GitLab 用户名后，展示该成员在团队所有项目上的综合城市视图（多项目叠加）。
  - 支持按项目拆分楼宇、按活动类型（编码、Review、Merge、Issue 运营）拆分城市分区。
  - 展示个人趋势类信息：活跃天数、典型协作者、核心项目等。

- **团队协作与健康度指标引入**
  - 在城市上叠加一些「协作信号」：如 Review 往来可以用光束/道路连接不同成员的大楼。
  - 提供基础看板指标：MR 通过率、平均 Review 时长、跨项目协作度、潜在孤岛项目/成员等。
  - 通过城市中冷清/暗淡区域提示「风险区域」：长时间没有更新、缺乏 Review、Bus Factor 过低等。

- **配置与权限控制**
  - 支持配置需要纳入统计的 GitLab Group/Project 列表，避免把实验性/一次性仓库混入城市。
  - 预留权限模型，与 GitLab 权限体系对齐，避免展示不该被看到的私有项目/数据。
  - 支持通过环境变量或管理面板配置访问 Token、采样周期、缓存策略等。

## Capabilities

### New Capabilities

- `team-gitlab-city-overview`：为整个团队构建一个统一的 GitLab 城市总览能力，聚合多 Group、多项目的数据，以城市形态展现整体活跃度、贡献分布和项目热度。
- `gitlab-project-sync`：从约 5 个 GitLab Group、20 个项目中稳定、增量地拉取 commit、MR、Review、Issue 等数据的同步能力，支持项目/Group 白名单配置与基础错误重试。
- `contribution-metrics-model`：定义并实现一套统一的「贡献度」与「活跃度」指标模型，将 commit 数量、变更规模、MR、Review、Issue 等行为综合打分，为城市高度/颜色等提供统一度量基础。
- `member-activity-visualization`：基于成员维度的可视化能力，将个人在各项目中的贡献汇总为城市中的楼宇/街区，支持时间范围切换和活动类型（编码/Review/运营）分层展示。
- `member-detail-dashboard`：输入 GitLab 用户名后，生成该成员的「个人 GitLab 城市」与概要面板，包括主参与项目、近期活跃度、协作伙伴分布与趋势曲线等。
- `team-segmentation-and-filtering`：按 Group、项目、技术栈、标签或自定义分组对团队进行切片的能力，对应城市中不同子区域和过滤视图，支持快速聚焦某一条业务线或小组。
- `realtime-city-refresh`：支持城市视图的实时/准实时刷新能力，通过轮询、Webhook 或定时任务等方式在合理的资源成本下保持城市与 GitLab 数据的同步。
- `collaboration-network-visualization`：通过在城市中绘制连接线/道路/光束等方式，呈现成员之间的 Review、MR 互动与协同关系，帮助识别协作枢纽与潜在孤岛。
- `risk-and-health-indicators`：在城市视图中对「长期不活跃仓库」「缺乏 Review 的 MR」「单人主导项目」等风险点进行标注的能力，以颜色、闪烁或特殊纹理提醒潜在健康问题。
- `config-and-permission-management`：提供城市构建相关的配置与权限管理能力，包含 GitLab Token/Group/Project 白名单配置、角色访问控制以及与现有认证体系的集成。

### Modified Capabilities

- `github-city-visualization`：原有 GitHub 单人/单仓库城市可视化能力需要扩展：
  - 从单仓库/本地 Git 日志的数据源切换或扩展为基于 GitLab API 的多项目汇聚。
  - 从「个人为主」的视角扩展为「团队总览 + 个人钻取」的多层视图结构。
  - 在可视化映射上，为新增的团队维度、协作关系、健康度指标预留对应视觉编码（颜色、纹理、布局）。

> 注：如现有 OpenSpec 中已经有覆盖 Git 城市可视化的 spec 名称，后续在 specs 阶段会用该实际名称替换此处占位。

## Impact

- **代码模块层面**
  - 现有城市渲染层：需要支持团队级城市布局、多 Group/项目分区、协作网络等新的可视化元素。
  - 数据采集与聚合层：需要新增 GitLab API 抽象层，支持多 Group/多项目配置、增量拉取与缓存。
  - 指标计算层：需要引入「贡献度/活跃度」计算逻辑，对不同类型的行为和不同项目进行统一加权。
  - 配置与权限模块：需要新增针对 GitLab Token、Group/Project 白名单、访问控制的配置能力。

- **外部系统与依赖**
  - GitLab：需要使用 Access Token、Webhook 或定时任务方式接入官方 API，需关注调用频率限制与权限边界。
  - 鉴权体系：如当前应用有登录/SSO 集成，需要与 GitLab 用户标识建立映射关系，以支持「按当前登录用户查看个人城市」等能力。

- **运维与资源**
  - 为数据同步与指标计算预留定时任务或后台 Job 能力，避免在前端请求路径上承担全部计算压力。
  - 根据团队规模和项目数量，为缓存与数据持久化做容量预估（例如近 90 天活动日志、指标快照等）。
  - 在可观测性层面，增加对数据同步失败、指标计算异常、GitLab API 限流等情况的监控与告警。

- **产品与使用体验**
  - 城市将成为团队内部「代码协作状态」的可视化入口，需要兼顾观赏性与可用性，避免过重的炫技导致信息噪音。
  - 需要为管理者、一线工程师和新成员提供不同层次的使用指引（如：如何解读楼高/亮度、如何发现风险区域等）。
