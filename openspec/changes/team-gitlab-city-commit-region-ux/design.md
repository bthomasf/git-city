# Commit 展示、区域可见性与楼栋点击体验 — 技术设计

## Context

- **现状**：团队 GitLab 城市已支持区域-小区-楼结构：city API 返回 `regions`、`buildingRows`（含 projectId、developerKey 等），commits API 按 projectId + username 返回最近 10 条 commit 及 `web_url`。团队页左侧为成员列表、右侧为选中成员/楼栋的详情；commit 列表当前展示「项目 #projectId」，侧栏内展示该楼 commit。地图上区域与小区仅通过建筑排布体现，无区域名、小区名等文字或边界标注；楼栋点击后依赖侧边面板展示 commit，面板收起时无法查看。
- **约束**：与现有 3D 场景（CityScene、InstancedBuildings）、团队页布局及 API 契约兼容；不破坏已有区域-小区配置与数据流。内网单机部署、中文界面。
- **干系人**：使用团队城市的内网用户。

## Goals / Non-Goals

**Goals:**

- 在团队页与楼栋点击后的 commit 展示中，用**项目名称**（或 path_with_namespace）替代项目 ID，便于识别所属项目。
- 在地图上**突出区域与小区**的视觉层次（区域标签、小区边界或名称），使用户一眼能分辨「有哪些区域、每个区域下有哪些小区」。
- **楼栋点击后的 commit 等详情**通过弹窗、浮动卡片或固定浮层展示，**不依赖侧边面板**是否展开；面板收起时仍可查看并关闭。

**Non-Goals:**

- 不改变区域-小区-楼的数据模型与配置结构。
- 不新增数据持久化或新后端服务；仅扩展 API 返回字段与前端展示方式。
- 建筑高度/亮窗等视觉规则、按用户筛选逻辑不在本设计范围内。

## Decisions

### 1. 项目名称的数据来源与展示

- **选择**：由**服务端在现有 API 中补充项目名称**供前端展示。具体：（1）commits API 已调用 `fetchProject(projectId)` 获取 `path_with_namespace`，在响应中为每条 commit 或整次响应增加 `project_name` 或 `path_with_namespace` 字段；（2）city API 的 `buildingRows` 若当前无项目名称，则在服务端构建 buildingRows 时按 projectId 解析一次项目信息（或从已有 GitLab 项目缓存/请求中获取），在每行增加 `projectName` 或 `pathWithNamespace`。前端团队页与 commit 列表统一使用该字段展示，不再显示「#projectId」为主。
- **理由**：项目名称与 path 由 GitLab 权威数据提供，避免前端为每个 projectId 再发请求；一次在 API 层补齐即可多处复用。
- **备选**：前端根据 projectId 再请求 project 接口 — 增加请求数与复杂度，不采用。

### 2. 区域与小区在地图上的可见性

- **选择**：在 3D 场景中增加**轻量级 2D 标注或地面标识**以体现区域与小区：（1）**区域**：在每块区域地块上方或边缘增加可读的**区域名称标签**（如 Three.js Sprite/Canvas 纹理或 DOM overlay），或在地面绘制区域边界色带/描边；（2）**小区**：在小区范围内或入口处增加**小区名称**标注，或与区域用不同底色/边界区分。实现上可复用现有场景相机与坐标，标签随视角缩放时可适当缩放或固定大小以保证可读性。
- **理由**：用户需求为「一目了然看到区域与小区」，文字标签与边界是最直接的方式；不改变建筑布局与实例化逻辑，仅增加一层标注或简单几何。
- **备选**：仅用颜色区分区域 — 无文字仍难以识别具体业务线名称，故需名称标注；纯 3D 立体字 — 实现与性能成本高，首版采用 2D 标签或 overlay 更稳妥。

### 3. 楼栋点击后 commit 详情的展示方式（不依赖侧栏）

- **选择**：点击某栋楼时，**在页面主体或 3D 容器外挂载独立展示层**（如 Modal、Popover 或固定位置的浮动卡片），用于展示「该楼对应 (project, developer) 的最近 10 条 commit」及可选的项目名称、跳转链接等。该展示层与侧边面板解耦：无论侧栏展开或收起，点击楼栋都会打开此展示层；关闭行为由用户点击关闭按钮或遮罩完成。若当前侧栏内已有「选中楼栋 + commit 列表」的区块，可保留用于「从成员列表选成员后」的右侧详情，与「从地图点击楼栋」的浮层并存，或统一为仅用浮层展示楼栋 commit（由实现时二选一）。
- **理由**：满足「面板收起时仍可查看楼栋 commit」；浮层/Modal 不依赖侧栏 DOM 结构，可挂载到 body 或根布局，避免被侧栏折叠影响。
- **备选**：强制展开侧栏再展示 — 改变用户当前布局习惯，不采用。仅在侧栏内展示 — 与「面板收起时也要能看」矛盾，不采用。

### 4. 前端展示项目名称的落点

- **选择**：团队页中所有出现「项目 #id」的地方改为优先展示项目名称（或 path_with_namespace），仅在不具备名称时回退为「#id」；commit 列表标题/副标题、成员详情中的「项目」列表等处统一使用名称。若 API 返回 `project_name` 与 `path_with_namespace` 两者，可优先显示 `project_name`，悬停或副标题展示 path 以区分同名项目。
- **理由**：与 proposal 的「更清晰」一致，减少对 ID 的记忆负担。
- **备选**：仅 commit 列表改 — 成员详情等处仍用 ID 会不一致，故统一改为名称。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 区域/小区标签过多导致画面杂乱或性能下降 | 标签数量等于区域数 + 小区数，可控；可做 LOD：远距离时只显示区域名，拉近再显示小区名。 |
| 浮层与 3D 场景的 z-index / 点击穿透 | 浮层使用 Portal 挂载到 body，遮罩拦截点击；3D 的 raycast 在打开浮层后可根据需要禁用或忽略楼栋点击直到关闭。 |
| 项目名称接口失败或延迟 | commits/city API 在无法获取项目信息时仍返回原有数据，名称字段为空则前端回退显示「#projectId」。 |

## Migration Plan

- 后端：先扩展 commits 与 city（buildingRows）的返回字段，增加项目名称或 path_with_namespace；保持现有字段兼容，前端可渐进切换展示。
- 前端：先改团队页与 commit 列表展示为「有名称用名称、无则用 ID」；再增加地图区域/小区标签与楼栋点击浮层；最后可选的清理：侧栏内楼栋 commit 区块与浮层二选一或统一。
- 无需数据迁移或停机；部署后即生效，回滚为还原前后端变更即可。

## Open Questions

- 区域/小区标签的最终视觉样式（字体大小、颜色、是否带背景）是否需与设计稿对齐，由实现阶段与产品确认。
- 楼栋点击后浮层与侧栏「选中楼栋」的交互是否完全统一为「仅浮层」展示 commit，避免两处重复逻辑，建议实现时定案。
