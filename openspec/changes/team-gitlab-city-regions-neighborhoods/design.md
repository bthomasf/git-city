# 团队 GitLab 城市（区域-小区-楼）技术设计

## Context

- **现状**：当前 GitLab 城市通过 `GITLAB_GROUP_IDS` / `GITLAB_PROJECT_IDS` 拉取项目与活动，`gitlab-city.ts` 将聚合后的成员指标转成「每成员一条」的 `DeveloperRecord`，再复用 `generateCityLayout` 得到「全城一版」的 buildings，即一个小区、每用户一栋楼。
- **约束**：内网单机部署、仅 GitLab 数据源、需与现有 3D 渲染管线（InstancedBuildings、CityScene、CityCanvas）兼容；团队侧希望中文界面与可配置视觉规则。
- **干系人**：使用团队城市的内网用户、运维（配置与部署）。

## Goals / Non-Goals

**Goals:**

- 配置模型支持「区域（业务线）→ 小区（项目）→ 楼（开发者）」层级；同一开发者在不同小区各有一栋楼。
- 全城展示为多区域、多小区、每小区多栋楼的 3D 布局。
- 支持按用户筛选：搜索后仅展示该用户在各区域各小区内的楼，其他楼隐藏。
- 点击某栋楼的某层时，展示该用户在该项目下的最新 10 条 commit。
- 建筑视觉规则（楼层高度、窗户亮灯等）通过配置驱动，可调公式/阈值。
- 团队城市入口与交互使用中文，适配内网办公场景。

**Non-Goals:**

- 不改造原有 GitHub 全球城市的数据模型与 API 契约（仅扩展 GitLab 团队城市路径与数据流）。
- 不做多租户或跨团队隔离（单团队、单内网部署）。
- 建筑定制、成就、支付、社交等能力不纳入本变更。

## Decisions

### 1. 配置结构：区域 → 小区 → 项目 ID

- **选择**：采用显式层级配置（如 YAML/JSON），结构为 `regions[]` → 每区域 `id/name` + `neighborhoods[]` → 每小区 `id/name` + `projectIds[]`。项目 ID 归属到小区，不再使用扁平的 `GITLAB_PROJECT_IDS` 作为唯一来源。
- **理由**：业务线（区域）与项目（小区）的对应关系由配置声明，便于按区域做布局与筛选；与「每小区多栋楼」的展示一致。
- **备选**：保留扁平 project ids 并增加「区域/小区」标签字段 — 不利于多区域布局与扩展，故不采用。

### 2. 建筑与开发者、项目的对应关系

- **选择**：每栋楼 = 一个「(项目, 开发者)」二元组；同一开发者在多个项目有贡献则在该多个小区各有一栋楼。楼层仍表示该开发者在该项目上的贡献强度（如 commit 数、MR 数等聚合指标）。
- **理由**：满足「多区域多小区多楼」与「按用户筛选时看到该用户在各小区的楼」的需求；与 GitLab 的 project-scoped 活动数据自然对应。
- **备选**：每用户全城仍一栋楼、用楼层区分项目 — 与当前 proposal 的「每小区多栋楼」不符，故不采用。

### 3. 城市数据 API 的返回形状

- **选择**：新增或扩展 GitLab 城市接口，返回结构为「区域列表 → 小区列表 → 建筑列表」，每建筑带 `regionId`、`neighborhoodId`、`projectId`、`developerKey`（如 username）、以及该 (project, developer) 的贡献指标与用于视觉规则的原始字段。按用户筛选时，可由同一接口加查询参数 `?filterUser=username` 在服务端过滤，或由前端根据全量数据过滤；优先服务端过滤以减小 payload。
- **理由**：前端需要按区域/小区做空间布局，并支持「全量」与「按用户」两种视图，服务端一次返回带层级信息的数据更清晰且可减少重复请求。
- **备选**：前端多次请求「按区域」「按小区」再拼装 — 增加请求数与复杂度，首版不采用。

### 4. 3D 布局与现有 CityLayout 的衔接

- **选择**：为团队城市单独实现「区域-小区-楼」的 2D 平面布局算法（每个区域一块地、区域内若干小区块、每小区内多栋楼按格点/网格排列），输出与现有 `generateCityLayout` 兼容的 `buildings`（含位置、尺寸、朝向）+ 扩展的 `regionId`/`neighborhoodId` 等元数据；河流/桥梁/装饰可简化或复用现有逻辑。不改为「每用户一栋」的现有 `generateCityLayout` 入参。
- **理由**：现有 layout 假设「一用户一建筑」且分区逻辑基于 language/district，与「(项目, 开发者) 一建筑」及配置驱动的区域不一致；独立布局函数便于迭代且不破坏 GitHub 城市。
- **备选**：强行把 (project, dev) 压成「虚拟用户」传入现有 layout — 分区与语义混乱，不采用。

### 5. 按用户筛选的交互与状态

- **选择**：搜索框输入用户名后，将「当前筛选用户」存入前端状态；请求城市数据时带 `filterUser=xxx`（或前端过滤）。场景中只渲染该用户在各区域各小区的楼，其他楼不渲染或设为完全透明/隐藏；提供「清除筛选」恢复全量视图。
- **理由**：与 proposal 的「只看到该用户在各区域各小区的楼」一致；实现上可为建筑列表过滤 + 保留相机/场景结构。
- **备选**：仅高亮该用户的楼 — 不满足「其他用户的楼隐藏」的明确需求。

### 6. 点击楼层与 Commit 列表

- **选择**：点击某栋楼（或某层）时，根据该建筑的 `projectId` + `developerKey` 请求「该用户在该项目下最近 10 条 commit」的接口（如 `GET /api/gitlab/commits?projectId=&username=&limit=10`），数据来自 GitLab Commits API；前端以弹窗或侧栏展示列表（标题、时间、message 等）。
- **理由**：GitLab 已有 commit 接口，按项目+作者过滤即可；10 条为固定需求，可配置为后续增强。
- **备选**：从现有 sync 结果里预取 — 需在聚合层持久化每条 commit，存储与刷新成本高，首版采用按需请求。

### 7. 视觉规则可配置（高度、亮灯）

- **选择**：引入「视觉规则配置」文档（如 JSON），定义：建筑高度与哪些指标、公式、区间映射；窗户亮灯比例/图案与哪些指标、阈值对应。应用层在生成单栋建筑的 `height`、`litPercentage` 等前读取该配置，按配置计算。配置可来自环境变量指向的 URL、或部署目录下的静态文件，默认内置一份与当前行为一致的 schema。
- **理由**：满足「可自由配置、不改代码」；与现有 `calcHeight`/亮窗逻辑解耦，便于后续调参。
- **备选**：写死在代码 — 不符合 proposal 的可配置要求。

### 8. 中文与内网部署

- **选择**：团队城市相关路由与组件使用中文文案（可集中在 `i18n` 或常量）；默认 `lang="zh-CN"` 或通过配置/环境变量切换；内网部署仅需保证 `GITLAB_BASE_URL`/Token 与区域-小区配置可用，无公网依赖。
- **理由**：与「公司内网、日常办公」场景一致；与现有 PROJECT-DESIGN 7.4 的 i18n 建议对齐。
- **备选**：全站改中文 — 若主站仍保留英文，则仅团队城市子路径中文即可，避免 scope 扩大。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 区域/小区过多导致建筑数量与 3D 性能下降 | 单次渲染建筑数上限、LOD 与实例化已有基础；可对区域/小区数量做配置上限或分页加载。 |
| GitLab API 限流或不可用 | 同步与按需 commit 请求均需限流与错误提示；必要时缓存 commit 列表短 TTL。 |
| 配置错误（错误 project id、错误区域层级）导致城市为空或错乱 | 配置校验（启动或定时）检查 project 可访问性与结构完整性；提供「配置检查」接口或日志。 |
| 与现有 GitHub 城市代码路径耦合 | 团队城市专用 API、layout 与页面组件与主城隔离；共享仅限 3D 组件（CityCanvas/InstancedBuildings）与类型定义。 |

## Migration Plan

1. **配置与兼容**：引入区域-小区配置格式与解析；保留现有 `GITLAB_PROJECT_IDS`/`GITLAB_GROUP_IDS` 仅作为「未配置区域时」的兜底来源，并输出迁移说明（如何从扁平列表改为区域-小区配置）。
2. **后端**：实现区域/小区/楼的数据聚合与城市 API 返回新结构；新增 commit 列表接口；增加视觉规则配置加载与应用。
3. **前端**：实现团队城市新布局生成、筛选用户状态与过滤渲染、楼层点击与 commit 弹窗/侧栏、中文文案与默认语言。
4. **验证**：内网部署后验证多区域多小区展示、筛选、点击 commit、修改视觉配置生效。
5. **回滚**：若采用特性开关，可关闭新布局与筛选逻辑回退到「单小区每用户一栋楼」；否则通过配置回退到仅使用扁平 project ids 的旧逻辑（需保留旧数据路径一段时间）。

## Open Questions

- 区域/小区在 3D 中的具体排布规则（如区域之间是否留主干道、小区块大小是否随建筑数自适应）需在实现时与产品确认。
- 视觉规则配置的默认 schema（字段名、公式表达式格式）是否需要在首版与现有 `calcHeight`/亮窗完全一致，以便 A/B 对比。
- 「点击某层」的交互是「整栋楼」还是「精确到层」：若精确到层，是否需要按层对应时间区间或 commit 子集，可留到实现阶段与交互稿确定。
