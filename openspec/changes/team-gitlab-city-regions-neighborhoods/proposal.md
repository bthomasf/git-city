# 团队 GitLab 城市：区域-小区-楼结构改造

## Why

当前 GitLab City 仅支持按 group 或项目 id 配置，城市呈现为「一个小区、每用户一栋楼」，无法表达「按业务线划分区域、区域内多项目（小区）、每项目多开发者（多栋楼）」的层级结构，也不支持按用户筛选视图、点击楼层查看 commit、以及建筑高度与亮灯等视觉规则的自定义。本次改造将配置与展示模型升级为「区域 → 小区 → 楼」，并支持用户维度的筛选与可配置视觉规则，以更好贴合团队内网部署与日常办公场景。

## What Changes

- **配置模型**：从「group / 项目 id」改为「区域（业务线）→ 小区（项目 id 列表）→ 楼（每栋楼对应一名在该项目有贡献的开发者）」；支持配置多个区域，每区域下配置多个小区（项目），每个小区内多栋楼（每开发者一栋）。
- **城市展示**：全城展示为「多个区域 × 多个小区 × 每小区多栋楼」的层级布局，而非单一小区、每用户一栋楼。
- **用户筛选**：提供按用户名/ID 搜索；搜索后仅展示该用户在各区域、各小区内的楼，其他开发者的楼隐藏。
- **楼层交互**：点击某一栋楼的某一层时，展示该用户在该项目下的最新 10 条 commit 记录。
- **视觉规则可配置**：楼层高度、窗户亮灯等与贡献/活跃度相关的展示规则通过配置项控制，支持按需调整公式或阈值，无需改代码。
- **BREAKING**：现有「仅 group/项目 id + 每用户一栋楼」的配置与展示方式将被新的区域-小区-楼模型替代，迁移需更新配置与数据结构。

## Capabilities

### New Capabilities

- `region-neighborhood-building-config`：区域（业务线）→ 小区（项目）→ 楼（开发者）的可配置层级结构；支持多区域、每区域多小区、每小区多栋楼；配置格式与校验规则。
- `user-filter-city-view`：按用户筛选城市视图；搜索用户后仅渲染该用户在各区域、各小区内的楼，其他楼隐藏；与全局城市视图的切换与状态管理。
- `floor-commit-detail`：点击具体楼层展示该用户在该项目下的最新 10 条 commit；数据来源、展示格式与交互（弹窗/侧栏等）。
- `city-visual-rules-config`：建筑视觉规则可配置（如楼层高度、窗户亮灯与贡献/活跃度指标的映射）；配置 schema、默认规则与运行时应用方式。

### Modified Capabilities

- （无：本变更在独立 change 下引入新能力，不修改现有 openspec 主 spec 的需求。）

## Impact

- **配置与数据**：新增/调整区域、小区、楼的配置结构；可能涉及 GitLab 项目与成员、贡献数据的聚合与缓存（按区域/小区/开发者维度）。
- **API**：城市数据接口需支持「按区域-小区-楼」的返回结构；可能新增或调整「按用户筛选」「单用户在某项目的 commit 列表」等接口。
- **前端 3D 与交互**：场景布局从「单小区多楼」改为「多区域多小区多楼」；搜索与筛选状态驱动可见建筑集合；楼层点击与 commit 详情展示组件；视觉规则从配置驱动（高度、亮灯等）。
- **部署与 i18n**：面向内网部署；若需整体中文交互，将影响文案与 i18n 配置（可在本 change 或后续 change 中处理）。
